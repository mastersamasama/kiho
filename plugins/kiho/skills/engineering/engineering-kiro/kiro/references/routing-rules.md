# 路由规则：模式选择

## 四模式决策矩阵

| 模式 | 何时使用 | 产出物 |
|------|---------|--------|
| **Feature** | 新功能、新页面、多文件变更、架构决策 | requirements.md → design.md → tasks.md |
| **Bugfix** | 缺陷修复、行为不符预期、回归问题 | bugfix.md → design.md(简版) → tasks.md |
| **Refactor** | 重构、技术债清理、模式迁移、性能优化 | requirements.md(轻量) → tasks.md |
| **Vibe** | 快速修复、小改动、单文件、配置调整 | 无规格文档，直接实现 |

---

## Feature 模式指标

满足 **1 个或以上** 即路由到 Feature：

1. **新功能或页面** — "实现"、"构建"、"创建"、"添加[功能]"、"新页面"
2. **多文件范围** — 变更将影响 3+ 文件或 2+ 目录
3. **数据库变更** — 新表、新列、迁移、schema 变更
4. **新 API 端点** — 新路由、新服务函数
5. **需求不明确** — 用户说"大概"、"可能"、"不确定"，范围模糊
6. **架构决策** — 需要在模式、库或方案之间做选择
7. **用户旅程** — 新的多步交互流程、表单流程
8. **跨切面关注** — 认证、缓存、错误处理等影响多模块的变更

**示例：**
- "实现餐厅评价页面" → Feature（新页面 + 多文件 + 新 API）
- "添加推送通知" → Feature（新功能 + 架构决策 + 跨切面）
- "重新设计过滤系统" → Feature（架构决策 + 多文件）

---

## Bugfix 模式指标

满足 **2 个或以上** 即路由到 Bugfix：

1. **Bug/缺陷关键词** — "bug"、"修复"、"坏了"、"不工作"、"报错"、"异常"
2. **行为不符预期** — "应该...但是..."、"预期...实际..."
3. **回归问题** — "之前可以，现在不行了"
4. **错误信息** — 用户提供了具体的错误信息或堆栈
5. **视觉缺陷** — "显示不对"、"样式错了"、"颜色不对"、"对齐问题"
6. **影响多文件** — 缺陷根因可能涉及多个文件

**示例：**
- "过滤器有时不工作" → Bugfix（bug 关键词 + 行为不符预期）
- "暗色模式下文字看不见" → Bugfix（视觉缺陷 + bug 关键词）
- "无限滚动偶尔加载重复数据" → Bugfix（行为不符预期 + 可能多文件）

**注意：** 如果缺陷根因不明确且可能涉及架构变更 → 升级为 Feature。

---

## Refactor 模式指标

满足 **2 个或以上** 即路由到 Refactor：

1. **重构关键词** — "重构"、"refactor"、"清理"、"优化"、"简化"、"拆分"、"合并"
2. **技术债** — "太复杂"、"难以维护"、"需要整理"
3. **模式迁移** — "迁移到新模式"、"统一为..."、"改用..."
4. **性能优化** — "太慢"、"性能差"、"需要优化"（不是 bug，是已知可改进点）
5. **不改变外部行为** — 内部改进，用户无感知
6. **代码组织** — "文件太大"、"拆分组件"、"移动到..."

**示例：**
- "重构 useExploreFilters hook，太复杂了" → Refactor
- "把所有硬编码颜色迁移到 CSS 变量" → Refactor（模式迁移）
- "优化首页加载性能" → Refactor（性能优化 + 不改变行为）

---

## Vibe 模式指标

满足 **2 个或以上** 即路由到 Vibe：

1. **简短描述** — 请求不超过 20 个词
2. **修复/补丁关键词** — fix、typo、tweak、adjust、rename、update、change、move、swap
3. **单文件范围** — 变更明确只影响一个文件
4. **配置变更** — 修改 config、env、package.json、tsconfig、lint/formatter 配置文件
5. **已知模式** — 用户引用现有模式："跟 X 一样"、"复制 Y 的模式"
6. **无新概念** — 无新数据模型、API、页面或组件

**示例：**
- "修复搜索栏 placeholder" → Vibe（简短 + fix 关键词 + 单文件）
- "更新卡片背景色" → Vibe（简短 + update 关键词 + 单文件）
- "重命名 getUserData 为 fetchUserProfile" → Vibe（修复关键词 + 已知模式）

## 边界情况

| 场景 | 路由 | 原因 |
|------|------|------|
| "在头部添加一个按钮" | Vibe | 单组件，范围清晰 |
| "添加带深度链接的分享按钮" | Feature | 新功能，需要设计决策 |
| "修复无限滚动的 bug" | Vibe 或 Bugfix | 简单 bug → Vibe；复杂根因 → Bugfix |
| "无限滚动有时显示重复数据" | Bugfix | 根因不明，需要调查 |
| "重构认证系统" | Refactor | 重构关键词，内部改进 |
| "重新设计认证流程" | Feature | 新用户旅程，非单纯重构 |
| "修复硬编码颜色" | Bugfix（少量） / Refactor（批量） | 2-3 处 → Bugfix；全局迁移 → Refactor |

---

## 覆盖规则

- 用户显式说 "直接做" / "quick fix" / "不用规格" → **Vibe**，无论指标
- 用户显式说 "规划一下" / "我想审查设计" → **Feature**，无论指标
- 用户提供 `--vibe` → **Vibe**
- 用户提供 `--bugfix` → **Bugfix**
- 用户提供 `--refactor` → **Refactor**
- 不确定时 → **默认 Feature** — 跳过规格步骤比补做未计划的工作成本更低

## 路由确认

**必须在路由后向用户确认：**

> "这看起来是 [Feature/Bugfix/Refactor/Vibe] 任务，因为 [原因]。是否按此模式继续，还是你希望使用 [其他模式]？"
