## Non-Goals

- **Not a replacement for Gate 17.** Gate 17 (novel contribution, v5.15) runs first as the pre-create similarity gate. v5.16 gates are about organization at scale, not pre-create dedup.
- **Not an LLM judge.** All six v5.16 gates are deterministic Python. The only LLM judge in skill-create is Gate 11 (transcript review), which v5.14 already separated evaluator from author.
- **Not a runtime permission system.** Capability verbs describe what a skill DOES, not what it's ALLOWED to do. Permissions stay at the agent + tool-allowlist level.
- **Not an embedding-based retrieval system.** kiho does not maintain a pre-loaded global embedding index (CLAUDE.md Non-Goal #3). Facet walking is deterministic, stateless, and explainable — the three properties embeddings don't have. Tier-3 on-demand embedding caches are permitted per-task with explicit eviction, not as a catalog-wide retrieval substrate.
- **Not a merge tool.** Gate 23 surfaces trigger-phrase collisions but does not automatically merge the colliding skills. Merge is an open problem (per v5.15 H4); humans drive, scripts assist.

## Architecture overview

v5.16 adds three primitives that compose into a two-stage retrieval pipeline:

```
QUERY
  │
  ├─ Primitive 1: walk CATALOG.md routing block (hierarchical, max depth 3)
  │   │
  │   ├─ top-level CATALOG (5 domains) → per-domain CATALOG → per-sub-domain CATALOG
  │   │
  │   ↓
  ├─ Primitive 2: filter by capability verb (closed 8-verb set)
  │   │
  │   ↓
  ├─ Primitive 3: filter by topic tag (controlled vocabulary)
  │   │
  │   ↓
CANDIDATE SET (MUST be ≤ 10)
  │
  ↓
Lexical scoring → top-3 results
```

Each stage prunes. At |S|=1000, filtering by `(capability, domain, topic_tag)` cuts the candidate set by ~100×. At |S|=38 (current kiho), it's ~10-20× typical.

## Primitive 1 — hierarchical walk-catalog

**Shape**: `skills/CATALOG.md` contains a `<!-- routing-block-start -->` / `<!-- routing-block-end -->` fenced YAML block with per-domain `routing-description` + `parent_of` lists. Domains that have been sub-grouped (currently `core`) also carry a `sub_domains:` nested block.

**Max depth**: 3. `top-level → domain → sub-domain → skill`. Authors **MUST NOT** nest deeper.

**Sub-grouping trigger**: a domain **SHOULD** split into sub-domains when it exceeds ~10 skills. `domain_fanout.py` (Gate 20 companion in a future v5.17) will enforce this; in v5.16 the split is manual and CEO-committee-approved.

**Current state (post-Stage D)**:
- `_meta/` — 8 skills (flat)
- `core/harness/` — 5 skills (kiho, kiho-spec, kiho-setup, kiho-init, org-sync)
- `core/hr/` — 2 skills (recruit, design-agent)
- `core/inspection/` — 3 skills (kiho-inspect, session-context, state-read)
- `core/knowledge/` — 3 skills (research, research-deep, experience-pool)
- `core/planning/` — 3 skills (kiho-plan, committee, interview-simulate)
- `kb/` — 8 skills (flat)
- `memory/` — 5 skills (flat)
- `engineering/` — 1 skill (flat)

**Regeneration**: `bin/catalog_gen.py` walks the tree and emits the per-domain tables; `bin/routing_gen.py` (post-hook) inserts/replaces the routing block. Both are idempotent.

## Primitive 2 — capability taxonomy

**Canonical verb set** (from `kiho-plugin/references/capability-taxonomy.md`):

```
create | read | update | delete | evaluate | orchestrate | communicate | decide
```

**Declaration**: every skill's frontmatter MUST contain `metadata.kiho.capability: <verb>`. One verb only.

**Classification rule**: primary effect. If a skill performs multiple capabilities, classify by the one that matches its reason for existing.

**Multi-capability skills**: if two capabilities are equally primary, the skill is probably doing too much. Split into sub-operations (like `skill-learn op=capture` / `op=extract` / `op=synthesize`), each with its own verb.

**Adding a verb**: the closed set is intentional. New verbs require a CEO-committee vote with a rationale citing 3+ pending skills. Expected frequency: <1/year.

**v5.16 catalog distribution** (post-Stage B):

| Verb | Count | Examples |
|---|---|---|

| read | 8 | skill-find, kb-search, memory-read, state-read, session-context, experience-pool |
| update | 6 | skill-improve, kb-update, kb-promote, memory-consolidate, org-sync, soul-apply-override |
| orchestrate | 5 | kiho, kiho-spec, kiho-plan, committee, engineering-kiro |
| evaluate | 3 | kb-lint, interview-simulate, evolution-scan |

## Primitive 3 — controlled vocabulary and faceted retrieval

**Vocabulary**: `kiho-plugin/references/topic-vocabulary.md` lists 18 approved topic tags. Every skill's `metadata.kiho.topic_tags` entries **MUST** come from this file.

**Tag count per skill**: 1-3 tags. Authors **SHOULD** pick the smallest number that discriminates the skill against its sub-domain siblings.

**Adding a tag**: CEO-committee vote with rationale citing 2+ pending skills. Expected frequency: 1-3 tags/quarter.

**Retrieval engine**: `skills/_meta/skill-find/scripts/facet_walk.py` implements the 5-step walk:

1. Tokenize query (stop-word removal).
2. Infer capability via keyword → verb mapping. First match wins. ~50 synonyms map to the 8 verbs (`create/draft/generate/author`, `find/search/lookup`, `update/improve/fix/patch`, etc.).
3. Infer domain by keyword overlap with routing-descriptions. Winner must beat runner-up by ≥2× to resolve.
4. Infer topic tags by exact match against the vocabulary (no stemming).
5. Walk routing block, intersect by applied facets, enforce ≤10 candidate-set ceiling.

## Gate 19 — routing-block sync

**Script**: `skills/_meta/skill-create/scripts/routing_sync.py`

**Tier**: error

**Checks**:
- Every skill ID in any `parent_of` list resolves to a real SKILL.md on disk (no ghosts).
- Every ACTIVE skill on disk appears in exactly one `parent_of` list (no orphans).
- Nested `sub_domains.X.parent_of` entries live at `skills/<domain>/X/<skill>/SKILL.md` on disk (no domain mismatch).
- Deprecated skills (lifecycle: deprecated OR `metadata.kiho.deprecated: true`) are not listed (no deprecated entries routed).

**Exit 0** when all four lists are empty. **Exit 1** on any finding.

**Runs at**: `skill-create` Step 3, `skill-deprecate` Step 6, and `bin/catalog_gen.py` post-hook.

## Gate 20 — capability declared

**Script**: `skills/_meta/skill-create/scripts/capability_check.py`

**Tier**: error

**Check**: frontmatter `metadata.kiho.capability` MUST be a single verb from the closed set.

**Modes**:
- `--skill <path>` — validate one SKILL.md (skill-create Step 3)
- `--all` — walk the catalog and audit every skill (migration verification)

**Exit 0** on pass. **Exit 1** when missing or out-of-set.

## Gate 21 — topic vocabulary

**Script**: `skills/_meta/skill-create/scripts/topic_vocab_check.py`

**Tier**: error

**Check**: every `metadata.kiho.topic_tags` entry MUST be in `kiho-plugin/references/topic-vocabulary.md`. Empty lists are rejected — every skill MUST declare at least one tag.

**Modes**: same as Gate 20.

## Gate 22 — candidate-set budget

**Script**: `skills/_meta/skill-create/scripts/candidate_set_budget.py`

**Tier**: error (primary attention gate)

**Check**: For each of the skill's trigger phrases (description + `## When to use` bullets that contain vocabulary discriminators), simulate `skill-find` facet walk with the skill's own capability + topic_tags forced as applied facets. Worst-case candidate set MUST be ≤10.

**Rationale**: the gate is asking "given this skill's declared facets, will the search narrow to ≤10 candidates for any of its trigger phrases?". Forcing the skill's own facets means we're testing discoverability using the author's discoverability claim — if they declared `capability=create, topic_tags=[authoring]`, the test simulates what happens when a query narrows to those facets.

**Exit 0** when worst-case ≤ 10. **Exit 1** otherwise with `status: underspecified`.

**Pipeline integration**: `candidate_set_budget.py` invokes `facet_walk.py` as a subprocess in `--gate-mode`. The two scripts share the facet-walk logic; Gate 22 is purely a test harness on top of it.

## Gate 23 — trigger-phrase uniqueness

**Script**: `skills/_meta/skill-create/scripts/trigger_uniqueness.py`

**Tier**: error

**Check**: Pairwise Jaccard on `when_to_use` / `trigger_phrases` unigrams (with STOP_WORDS filtering). **Block** at Jaccard ≥ 0.70. **Warn** at 0.50 ≤ Jaccard < 0.70.

**Why stricter than Gate 17**: Gate 17 checks description Jaccard with a 0.60 block threshold. Trigger phrases are literal — agents select by exact-match on trigger text, so collisions have immediate routing impact. Fixing a shipped skill's trigger phrase is a bigger migration than rewriting its description.

**Modes**:
- `--draft <path>` — check a draft against every existing skill
- `--all` — pairwise audit of the whole catalog

## Gate 24 — agent portfolio density

**Script**: `skills/_meta/skill-create/scripts/agent_density.py`

**Tier**: warn

**Check**: For every agent in `agents/*.md`, aggregate the `skills:` frontmatter array by capability and by domain. Thresholds:
- **per-capability**: warn at ≥5, error at ≥8
- **per-domain**: warn at ≥8, error at ≥12

**Rationale**: the arXiv §5.2 plateau applies at invocation time too. An agent whose portfolio clusters on one capability or domain hits the same selection-accuracy cliff even if the global catalog is clean.

**Current findings** (Apr 2026):
- kiho-ceo: 13 skills, domain max `core=8` → warn (dom_warn=8)
- kiho-kb-manager: 8 skills, domain max `kb=8` → warn
- All others: pass

## Failure playbook

Decision trees for each v5.16 gate when triggered:

### Gate 19 — routing sync fails

```
routing_sync result
   │
   ├─ ghost_entries[] non-empty       → Route 19-A: remove ghost IDs from routing block
   ├─ orphan_skills[] non-empty       → Route 19-B: add skill to correct parent_of via routing_gen.py
   ├─ domain_mismatch[] non-empty     → Route 19-C: move skill directory OR update parent_of
   ├─ deprecated_listed[] non-empty   → Route 19-D: remove deprecated IDs from parent_of
   └─ all empty                        → PASS
```

### Gate 20 — capability missing or out-of-set

```
capability verdict
   │
   ├─ missing field                    → Route 20-A: prompt author to pick one of 8 verbs
   ├─ out-of-set verb                  → Route 20-B: script suggests closest canonical via keyword match
   └─ genuinely new verb needed        → Route 20-C: escalate to CEO committee (rare, <1/year)
```

### Gate 21 — tag not in vocabulary

```
topic_vocab verdict
   │
   ├─ tag not in vocab                 → Route 21-A: script suggests closest vocab entry via Jaccard
   └─ vocab expansion warranted        → Route 21-B: escalate to CEO committee with rationale
```

### Gate 22 — worst-case candidate set > 10

```
candidate_set_budget verdict
   │
   ├─ worst_case 11-20                 → Route 22-A: add a discriminating topic tag (narrow the facet)
   ├─ worst_case 21-50                 → Route 22-B: capability may be wrong; reconsider the verb choice
   └─ worst_case > 50                  → Route 22-C: go to Gate 17 — the problem is genuine similarity, not under-specification
```

### Gate 23 — trigger phrase collision

```
trigger_uniqueness verdict
   │
   ├─ Jaccard ≥ 0.70 against one skill → Route 23-A: rewrite conflicting phrase with discriminator
   ├─ collision is genuine             → Route 23-B: redirect one skill to skill-derive (specialization)
   └─ CEO-approved override            → Route 23-C: --force-collision (logged to skill-invocations.jsonl)
```

### Gate 24 — agent density over threshold

```
agent_density verdict
   │
   ├─ per-capability warn/error        → Route 24-A: remove rarely-used skills (telemetry-driven)
   ├─ per-domain warn/error            → Route 24-B: split the agent into two specialized agents
   └─ hot spot is Gate 19 fault        → Route 24-C: cascade to Gate 19 / Primitive 1 restructure
```

## Worked examples

Each example shows a query → facet inference → candidate set → top result, end-to-end against the current 38-skill catalog. All inputs and outputs are byte-identical with what `skills/_meta/skill-find/scripts/facet_walk.py` actually emits when run with the shown command line.

### Example 1 — Update-with-domain-resolution (facet walk narrows to 2)

**Command**:
```bash
python skills/_meta/skill-find/scripts/facet_walk.py \
    --query "I need to update an existing skill to fix a bug in its body" \
    --limit 3
```

**Expected output** (abbreviated):
```json
{
  "passed": true,
  "status": "ok",
  "applied_facets": {
    "capability": "update",
    "domain": "kb",
    "topic_tags": []
  },
  "candidate_count": 2,
  "top_results": [
    {"id": "sk-014", "name": "kb-update", "score": 0.2, ...},
    {"id": "sk-018", "name": "kb-promote", "score": 0.0, ...}
  ]
}
```

**What happened**: "update" → capability=update (deterministic verb match). "update" also appears in kb's routing-description ("update pages"), so domain=kb wins with 1 overlap vs 0 for all others. The intersection is 2 kb skills. Good enough (≤10 ceiling).

**Why this is NOT a bug**: even though the user probably meant `skill-improve` (which is `capability: update, topic_tags: [authoring, lifecycle]`), the query didn't contain "authoring" or "improve" as tokens. The facet walker is deterministic — when ambiguous, it commits to whichever facet wins. The `--domain _meta` override flag is the escape hatch when the author knows the target is in a different domain.

### Example 2 — Research query with multi-tag inference (candidate set 2)

**Command**:
```bash
python skills/_meta/skill-find/scripts/facet_walk.py \
    --query "research a topic thoroughly using external documentation" \
    --limit 3
```

**Expected output**:
```json
{
  "passed": true,
  "status": "ok",
  "applied_facets": {
    "capability": null,
    "domain": "core",
    "topic_tags": ["research"]
  },
  "candidate_count": 2,
  "top_results": [
    {"id": "sk-010", "name": "research", "score": 0.4, "capability": "read", ...},
    {"id": "sk-rdp", "name": "research-deep", "score": 0.4, "capability": "create", ...}
  ]
}
```

**What happened**: "research" matches the `research` tag in the vocabulary. "core" wins domain by keyword overlap with `core`'s routing-description. No capability verb matched the query tokens (the word "research" is a noun here, not in CAPABILITY_KEYWORDS). The intersection is 2 skills (experience-pool has the `learning` tag instead of `research` and is excluded).

### Example 3 — Underspecified vague query (gate mode fails fast)

**Command**:
```bash
python skills/_meta/skill-find/scripts/facet_walk.py \
    --query "handle the thing" \
    --limit 3 --gate-mode
```

**Expected output**:
```json
{
  "passed": false,
  "status": "underspecified",
  "applied_facets": {
    "capability": null,
    "domain": null,
    "topic_tags": []
  },
  "candidate_count": 38,
  "hard_ceiling": 10,
  "narrowing_hints": ["capability", "domain", "topic_tag"],
  "message": "Candidate set is 38 > ceiling 10. Add one of these facets to narrow: ['capability', 'domain', 'topic_tag']"
}
```

**Exit code**: 1 (policy violation — under-facet-specified in gate mode).

**What happened**: "handle" and "thing" match no keyword mappings. All three facets are null. The candidate set is the whole catalog (38). In `--gate-mode`, this fails with actionable narrowing hints instead of returning garbage.

## Rejected alternatives

### A1 — LLM judge at Gate 22

**What it would look like.** Replace the deterministic facet walk with a sub-agent that reads the draft + the whole catalog and decides whether the skill is discoverable. Gate 22 spawns a judge instead of simulating a mechanical walk.

**Rejected because.**
- **Non-determinism.** The same draft evaluated twice produces different verdicts. CI reproducibility breaks.
- **Token cost.** ~30k tokens per judge call vs ~500 tokens for the facet walk subprocess.
- **Explainability loss.** A judge can't tell the author *which facet would narrow*. The deterministic walker emits `narrowing_hints: ["topic_tag"]` precisely.
- **Anthropic precedent.** Gate 11 (transcript review) is the only LLM-judge gate in skill-create; the rest are deterministic per Anthropic's "Demystifying Evals" guidance. Gate 22 follows the same rule.

**Source:** Anthropic "Demystifying Evals for AI Agents" Jan 2026 §step 7 — deterministic gates should stay deterministic.

### A2 — Embedding-based facet retrieval

**What it would look like.** Replace the keyword-matching facet inference with sentence-transformers cosine similarity against per-verb / per-tag embeddings. The facet walker computes `similarity(query, verb_embedding)` for each verb and picks the max.

**Rejected because.**
- **Daemon requirement.** Embeddings need a running model server. kiho does not ship an MCP server (CLAUDE.md Non-Goal #2) and does not maintain a pre-loaded global embedding index (Non-Goal #3).
- **Model-version drift.** Two different sentence-transformer versions produce different cosine scores, breaking reproducibility across CI environments.
- **Gate 22 would become stochastic.** Worst-case candidate set would vary between runs, defeating the ≤10 ceiling as a hard invariant.

**Source:** kiho CLAUDE.md Non-Goal #3 (no embedding-based retrieval); v5.15 H5 research findings (pnpm / cargo / Bazel all use deterministic text scanning for the same reason).

### A3 — Depth-4 hierarchy (top-level → domain → sub-domain → sub-sub-domain → skill)

**What it would look like.** Allow hierarchy beyond max depth 3. A future `core/knowledge/research/external/arxiv-crawler` skill would live at depth 4.

**Rejected because.**
- **Walk-load problem.** Each level the agent traverses is another attention beat. Max depth 3 gives 3 pages × 20 entries = 60 items loaded to reach any skill. Depth 4 doubles that.
- **Python stdlib / man pages precedent.** Both cap hierarchy at ~3 levels for the same reason.
- **Sub-grouping vs new domain.** If a sub-domain grows past ~10 skills, the right response is promoting it to a top-level domain, not nesting deeper.

**Source:** Python standard library module layout; FHS man page section convention; Homebrew tap structure.

### A4 — Raise Gate 22 ceiling to 15 or 20

**What it would look like.** `candidate_set_budget.py --hard-ceiling 15` or `--hard-ceiling 20`. Loosens the attention-budget constraint.

**Rejected because.**
- **arXiv §5.2 plateau at |S|≤20.** 10 is already past the midpoint of the safe range. 15 approaches the inflection; 20 is at the cliff edge.
- **Migration headroom.** 38/38 existing skills pass at ceiling=10 after Stage B+C annotation. Raising would be a false safety net.
- **Plan explicit threshold.** v5.16 plan says "If Gate 22 blocks >20% of new drafts during the first month, loosen to ≤15 and document the change. Do NOT switch to token-counting." Current blocking rate on migration was 0% after the topic-tags-force fix; no trigger to loosen.

**Source:** arXiv 2601.04748 §5.2; kiho v5.16 plan Risks section.

## Future possibilities (non-binding)

> **Non-binding note (Rust RFC 2561 convention):** *"Having something written down in this section is not a reason to accept the current or a future RFC; such notes should be in the section on motivation or rationale in this or subsequent specs."* The items below are hints, not commitments.

### F1 — Stemming in domain-keyword inference

**Sketch.** Currently `infer_domain` does exact token matching against each domain's routing-description. Add a Porter stemmer (stdlib `re` with hand-rolled suffix rules, no PyPI dep) so "improvement" in the routing-description matches "improve" in the query.

**Not a commitment.** Current Example 1's domain-mismatch could be fixed by stemming, but stemming introduces its own false-positive risks. Measure first.

### F2 — Multi-facet disambiguation UI

**Sketch.** When `candidate_count` is exactly 2-5 (narrow but not resolved), emit a ranked list with per-skill facet contributions so the caller can pick manually rather than relying on lexical scoring alone. This is a runtime enhancement to `skill-find`, not a Gate 22 change.

**Not a commitment.** Current lexical scoring within the narrow set works in practice; this is a nicety for human invocation, not agent invocation.

### F3 — Domain pre-filter confidence threshold

**Sketch.** The `infer_domain` function currently picks the winner only if it beats runner-up by ≥2×. At low confidence (e.g., 1 vs 0 overlap), fall through to no-domain-filter instead of committing. This would have fixed Example 1's domain mismatch.

**Not a commitment.** Measure empirical mismatch rate first. The 2× threshold is tunable.

## Exit-code convention (for the scripts backing these gates)

All Python scripts under `skills/_meta/skill-create/scripts/` and `bin/` referenced by v5.16 gates follow the kiho exit-code convention documented in `kiho-plugin/references/skill-authoring-standards.md`:

- **0** — success (gate passed, query resolved, operation complete)
- **1** — policy violation (gate failed, Jaccard over threshold, candidate set over ceiling, out-of-set verb, etc.)
- **2** — usage error (CATALOG.md missing, argument missing, file not found)
- **3** — internal error (unhandled exception; caller should check stderr)

Specifically for v5.16 facet retrieval scripts:

| Script | 0 | 1 | 2 | 3 |
|---|---|---|---|---|
| `facet_walk.py` | candidates returned / underspecified in non-gate mode | underspecified in `--gate-mode` | CATALOG missing, query missing | internal error |
| `candidate_set_budget.py` | worst_case ≤ ceiling | worst_case > ceiling | draft path missing, facet_walk.py missing | internal error |
| `routing_sync.py` | all lists empty | any ghost/orphan/mismatch/deprecated | CATALOG missing, routing block missing | internal error |
| `capability_check.py` | all checked skills pass | any out-of-set verb | SKILL.md missing, taxonomy file missing | internal error |
| `topic_vocab_check.py` | all checked skills pass | any out-of-vocab tag OR stale outer metadata.topic_tags | SKILL.md missing, vocab file missing | internal error |
| `trigger_uniqueness.py` | no pairs ≥ block threshold | any pair ≥ block threshold | SKILL.md missing | internal error |
| `agent_density.py` | all agents within warn thresholds | any agent ≥ error threshold | agents dir missing | internal error |

Full convention: `kiho-plugin/references/skill-authoring-standards.md` v5.15.2 §"Exit-code convention".

## Grounding

> **arXiv 2601.04748 §5.2 (Single-Agent with Skills, Apr 2026):** *"At small scales (|S| ≤ 20), accuracy remains above 90%, but degrades steadily beyond |S| = 30, falling to approximately 20% at |S| = 200."*

Used by: Gate 22 ceiling calibration (10 gives 2× headroom from the |S|=20 plateau). `skill-find` facet walk motivation.

> **arXiv 2601.04748 §5.3:** *"At identical |S| = 20, replacing unique skills with base-competitor pairs causes an 18-30% accuracy drop. This demonstrates that semantic structure determines selection difficulty."*

Used by: rejected alternative A1 (raising ceiling to 15). Semantic confusability dominates skill count.

> **arXiv 2603.02176 (AgentSkillOS) §2.1.1:** *"If the number of skills in any generated category after Skill Assignment is equal to 1, that category and all its skills will be merged into the most relevant target category."*

Used by: Primitive 1 sub-domain split rule (sub-group when domain > 10 skills).

> **Kubernetes API verbs** (get/list/watch/create/update/patch/delete) — https://kubernetes.io/docs/reference/access-authn-authz/authorization/

Used by: Primitive 2 closed verb set design. kiho's 8 verbs are a minor extension (adds `orchestrate`, reserves `decide`).

> **Library of Congress classification** — https://www.loc.gov/catdir/cpso/lcco/

Used by: Primitive 3 controlled vocabulary pattern. The anti-drift rationale in `topic-vocabulary.md` cites this.

Full research archive: `kiho-plugin/references/v5.16-research-findings.md`.
