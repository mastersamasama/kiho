# kiho-setup

First-run scaffolder for kiho. Ensures a valid `company_root`, then idempotently creates the project and company state trees from templates.

## Overview

kiho has two knowledge-base tiers: a **project KB** under each project's `.kiho/` and a **company KB** under `$COMPANY_ROOT/` (user-global). This skill creates both trees from templates and records the user's company-root choice in `config.toml` (migrated from YAML in v5.19.3).

## Procedure

### 1. Resolve `company_root`

Read `${CLAUDE_PLUGIN_ROOT}/skills/core/harness/kiho/config.toml`.

If `company_root` is non-empty, skip to step 2.

If empty or missing, call `AskUserQuestion` with this exact question and options:

> **Where should kiho store the company-level knowledge base?**
> kiho keeps a project KB per repo and a company KB that follows you across projects. Pick a stable location outside any project.

- `D:/Tools/kiho/` — Recommended for Windows; outside any project
- `~/kiho/` — Recommended for Unix; home directory
- Inside `.claude/kiho-company/` — Colocated with other Claude Code data
- Custom — any absolute path

Validate the answer:
- Must be absolute
- Must be writable
- Must not be inside any `.kiho/` directory

Write the answer back to `config.toml` using `Edit` (preserve the rest of the file). TOML string syntax is `company_root = "D:/Tools/kiho/"` — double-quoted, forward slashes on Windows OK.

### 2. Scaffold the company tree

See `references/scaffold-tree.md` for the full directory layout.

Create `$COMPANY_ROOT/` from the company templates. Files to copy:
- `knowledge-base.md` ← `templates/kb-company-knowledge-base.template.md`
- `rules.md` ← `templates/kb-company-rules.template.md`
- `memos.md` ← `templates/kb-company-memos.template.md`
- 12 empty index shells (see `references/scaffold-tree.md` for the full list and shell format)

Run template substitution: replace `{{company_root}}`, `{{user_name}}`, `{{iso_timestamp}}` before writing.

### 3. Scaffold the project tree

Determine the project root via `pwd`. Create `<pwd>/.kiho/` from the project templates. Files to copy:
- `kb/knowledge-base.md` ← `templates/kb-knowledge-base.template.md`
- `kb/rules.md` ← `templates/kb-rules.template.md`
- `kb/memos.md` ← `templates/kb-memos.template.md`
- `state/org.json` ← `templates/org-bootstrap.json` **(deprecated — see org-registry.md below)**
- `state/org-registry.md` ← `templates/org-registry.template.md` (replaces org.json as the canonical org state)
- `state/capability-matrix.md` ← `templates/capability-matrix.template.md`
- `state/management-journals/` ← empty directory (one .md per department leader)
- `state/skill-invocations.jsonl` ← empty file (tracks skill usage per iteration)
- `state/agent-performance.jsonl` ← empty file (tracks per-agent task outcomes)
- `state/ceo-ledger.jsonl` ← empty file (CEO per-iteration decision / action ledger read by INITIALIZE step 12 and LOOP step a; tail-200 is the hot read path)
- `state/cross-agent-learnings.jsonl` ← empty file (queue drained by `memory-cross-agent-learn`; INITIALIZE step 15 reads pending rows)
- `state/plan.md` ← `templates/plan.template.md`
- `state/AGENT.md` ← `templates/AGENT.template.md`
- `state/completion.md` ← `templates/completion.template.md`
- `agents/ceo-01/memory/lessons.md` ← empty with a single `# CEO lessons` header; INITIALIZE step 9 reads last 5 entries
- `agents/ceo-01/memory/todos.md` ← empty with a single `# CEO todos` header; INITIALIZE step 9 reads non-archived rows
- `agents/ceo-01/memory/.last-reflect` ← touch with current ISO-8601 UTC timestamp; INITIALIZE step 14 checks this for self-reflect cadence
- `CONTINUITY.md` ← `templates/CONTINUITY.md`
- All 12 project-tier index shells
- All empty directories (`state/briefs/`, `committee/`, `specs/`, `agents/`, etc.)

**v5.21 additions (cycle-runner, inboxes & queues):**
- `state/cycles/` ← empty directory; `cycles/INDEX.md` is regenerated by `bin/cycle_index_gen.py` at CEO DONE — not seeded here
- `state/inbox/` ← empty directory (one `<agent_id>.jsonl` created lazily by `memo-send` on first peer-to-peer write)
- `state/handoffs/` ← empty directory (one `<turn_id>.jsonl` per turn from `handoff-accept`; rotates to `handoffs/archive/` after CEO DONE)
- `state/feedback/requests.jsonl` ← empty file (proposed user-feedback prompts queued by `user-feedback-request`)
- `state/feedback/responses.jsonl` ← empty file (drained by CEO; analyzed by `memory-reflect`)
- `state/commit-ceremony/` ← empty directory (per-target `<target_canonical_ref>/pending.jsonl` + `archive/` created lazily by callees of `commit-ceremony`)
- `state/integrations/registry/` ← empty directory (one `<integration_id>.md` per registered MCP / native tool / CLI; `integration-register` writes via `storage-broker` per-entry; per-file template at `templates/integration-entry.template.md` referenced by the skill, not copied here)
- `state/integrations/debt.jsonl` ← empty file (one row per registered-but-never-called integration; cleared lazily by `integration-audit` on first telemetry hit)
- `state/values-flags/` ← empty directory (one `<flag_id>.md` per CEO ruling produced by `values-flag`)

These align with `references/data-storage-matrix.md` §1 (`values-flags`), §3 (`integrations-registry`), §10c (`memo-inbox`, `handoff-receipts`, `feedback-queue`, `commit-ceremony-pending`), and §10b (`cycle-*`).

Template substitutions: `{{project_name}}`, `{{project_slug}}`, `{{iso_timestamp}}`.

### 3b. Migrate deprecated org.json

If `.kiho/state/org.json` exists, migrate its contents to `org-registry.md` format:
1. Read `org.json` and extract agents, departments, and hierarchy
2. Write the equivalent markdown structure into `org-registry.md` (if `org-registry.md` does not already exist)
3. Do NOT delete `org.json` — leave it in place for backward compatibility, but all new reads should use `org-registry.md`
4. Log the migration in the setup receipt under `migrations`

### 4. Idempotency

**Never overwrite an existing file.** For each target path:
- If the file exists and is non-empty, skip and record it in the `skipped` list.
- If the file is missing, create it from the template.
- If the file exists but is empty (0 bytes), replace it from the template.

### 5. Return a receipt

Emit this structured summary to the caller:

```
## kiho-setup receipt
status: ok | partial | error
company_root: <resolved path>
project_kiho: <project>/.kiho/
created_files: [<list of newly-created paths>]
skipped: [<list of paths that existed and were kept>]
errors: [<list of error messages with paths>]
```

On any permission error, disk-full error, or invalid-path error, set `status: error` and return immediately without retrying.

## Anti-patterns

- Do not run on every `/kiho` invocation. Only run when `company_root` is empty or the user explicitly requests setup.
- Do not install Python packages, create `.gitignore` files, modify git state, or run network operations.
- Do not prompt for any configuration beyond `company_root`. Other values have defaults.
- Do not overwrite existing content. Idempotent create-only.
- Do not skip the company-root validation — a wrong path corrupts the whole skill system.
