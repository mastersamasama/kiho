## Headline findings

### H1. Token budget is a proxy; attention is the metric

arXiv 2601.04748 §5.2 measures LLM skill-selection accuracy vs catalog
size under flat selection:

> "At small scales (|S| <= 20), accuracy remains above 90%, but degrades
> steadily beyond |S| = 30, falling to approximately 20% at |S| = 200."
> — arXiv 2601.04748 §5.2

And §5.3 on semantic confusability:

> "At identical |S| = 20, replacing unique skills with base-competitor
> pairs causes an 18-30% accuracy drop. This demonstrates that semantic
> structure determines selection difficulty."
> — arXiv 2601.04748 §5.3

The failure mode at scale is **the agent cannot decide which skill to
pick**, not **the context overflowed**. Token count is a proxy: a 3k-token
skill in a 12-member cluster is unselectable; a 6k-token skill that
cleanly discriminates is trivially selectable. v5.16 replaces Gate 3
(body token count, demoted to warn) with Gate 22 (candidate-set budget)
as the primary attention gate.

### H2. Two-stage retrieval is the convergent pattern across mature ecosystems

Every mature catalog-of-many-things system uses two-stage retrieval: a
coarse facet filter reduces the candidate set from N to ~10, then a fine
scorer picks within the filtered set.

- **npm / PyPI**: flat package namespace, multi-facet discovery (name +
  keywords + categories + owner + downloads + full-text search).
- **Kubernetes API**: `(apiGroup, resource, verb)` filter before any
  object touch. 7 canonical verbs (`get/list/watch/create/update/patch/delete`).
- **Library of Congress / arXiv / WordNet**: controlled vocabulary +
  committee-gated additions.
- **Homebrew / man pages / Python stdlib**: walkable tree for humans +
  keyword index for machines.

kiho v5.16 adopts:

- **Primitive 1** — hierarchical walk-catalog (Homebrew/stdlib pattern)
- **Primitive 2** — closed 8-verb capability taxonomy (Kubernetes pattern)
- **Primitive 3** — controlled topic vocabulary + faceted retrieval
  (Library of Congress pattern)

### H3. Hierarchical routing scores ~4× higher than flat at |S|=200

arXiv 2603.02176 (AgentSkillOS) §2.1.1 measures Bradley-Terry pairwise
win rates for skill selection at |S|=200:

> "If the number of skills in any generated category after Skill Assignment
> is equal to 1, that category and all its skills will be merged into the
> most relevant target category."
> — arXiv 2603.02176 §2.1.1

Bradley-Terry scores: hierarchical 100.0 vs flat 24.3 at |S|=200.

kiho v5.16 mandates sub-domain splitting when a domain crosses ~10 skills
(currently applied to `core` only, which was at 16 and past the plateau).
`core` → `harness/`, `hr/`, `inspection/`, `knowledge/`, `planning/`.

### H4. Closed verb sets beat open verb sets

Kubernetes (7 verbs), HTTP REST (4-5), SQL DML (4). More verbs hurt
discrimination (the classifier can't reliably bucket); fewer verbs hurt
selectivity (too many distinct operations merge).

kiho v5.16 settles on 8 verbs: `create | read | update | delete |
evaluate | orchestrate | communicate | decide`. Rationale for 8 rather
than 7:

- Kubernetes has no `orchestrate` — it's a CRUD-plus-watch API. Agent
  skills need orchestration as a distinct primary effect.
- The 8th slot (`decide`) is reserved for committee-gated decision
  skills. It has one pending consumer today (committee, which is
  currently classified as `orchestrate`) and is expected to grow.

### H5. Controlled vocabulary is the only scaling mechanism for tags

Library of Congress, arXiv, WordNet, Dewey Decimal, MeSH — every mature
taxonomy project converges on controlled vocabularies after trying
free-form tags and failing. The failure mode of free-form tags is drift:
`kb-retrieval` vs `knowledge-lookup` vs `fetch-kb` mean the same thing
but don't match, so facet filtering produces a spurious empty set.

v5.16 seeds 18 controlled topic tags. Growth is committee-gated at
~1-3 tags/quarter. The vocabulary file carries a `## Non-Goals` section
with the anti-drift rationale.

## Rejected alternatives

### A1. Raise Gate 22 ceiling to 15

Tempting when 6/38 skills failed the initial run (16% blocking rate,
close to the plan's 20% intervention threshold). Rejected because:

1. The 10 ceiling has 2× headroom from arXiv §5.2's |S|≤20 plateau.
2. The failures were all resolved by forcing the skill's own declared
   capability + topic_tags as facets (the test methodology fix) — not
   by loosening the ceiling.
3. Raising the ceiling weakens the attention-budget argument. 15 is
   past the plateau inflection.

### A2. Embedding-based retrieval (sentence-transformers for skill similarity)

Rejected for the same reason as v5.15 H5: requires a daemon. kiho is
markdown-only (CLAUDE.md Non-Goal #2). Embedding indexes drift across
model versions. Gate 22 + facet walking is the deterministic,
markdown-compatible alternative.

### A3. Nested topic-tag hierarchy

Rejected because flat controlled vocabularies are the convergent
pattern. Hierarchical tags (e.g., `authoring.skill.creation` vs
`authoring.skill.deprecation`) add a parsing layer every script must
handle, defeat the facet-walk's O(1) lookup, and don't meaningfully
improve selection at 18 tags.

### A4. Automated capability-verb inference from skill body

Rejected because the verb is a declarative commitment by the author,
not a property that should be inferred. Inference would make the
capability field non-authoritative and weaken Gate 20. The migration
script uses keyword heuristics as a first pass, but every classification
is human-reviewed.

### A5. Weekly audit as a pre-registration gate

The orphan / stale-draft / confusability checks were originally slated
as Gates 23-25 in the initial v5.16 draft. Demoted to a weekly cron
audit because:

1. Orphans are rare and lazy cleanup is fine.
2. Stale DRAFTs accumulate over months, not during a single skill-create
   run.
3. Confusability drift is a slow-moving metric; per-draft checks are
   Gate 17's job.

Consolidated into `bin/catalog_walk_audit.py` as a kb-manager weekly
task.
