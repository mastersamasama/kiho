# 99 — v5.23 adoption roadmap (synthesis)

## Executive summary

Six committees closed unanimously at confidence ≥ 0.90, all within ≤ 2 rounds. Key pattern: five of six rejected the "add a new skill portfolio" reflex in favor of minimal extensions to existing infrastructure. Total new kiho skills proposed across all six committees: **three** (all in committee 01 OKR). Total new Python helpers: **four**. Total new data-storage-matrix rows: **four** (all Tier-2 regenerable except one T1 announcements row). Zero new agent roles. Zero new hook scripts (committee 02 refactors two existing hooks; no new ones).

v5.23 can ship as a much smaller release than v5.22. The committees collectively demonstrated a mature kiho heuristic: **OA's product-shaped features decompose into thin reference-file additions + small scripts on top of existing primitives**, rather than parallel infrastructure.

## Recommended adoption ordering

Two criteria dominate: (a) unblocks downstream work, (b) low cost + high value ratio.

### Phase A — quick wins (target: days 1–2)

**A1. Committee 04 — pulse surveys (reject + reinforcements)**

- Edit `references/committee-rules.md` to document lightweight-committee variant.
- Edit `skills/core/ceremony/retrospective/SKILL.md` to add process-friction question.
- Ship `bin/pulse_aggregate.py`.

Cost: ~1h. Risk: negligible. Value: closes the "should we build pulse?" question durably + gets process-friction signal into retrospectives immediately.

**A2. Committee 03 — broadcast (minimal extension)**

- Add wildcard recipient support to `memo-send`.
- Add announcement re-surface to `shift-handoff`.
- Introduce `.kiho/state/announcements/` T1 data class (Storage-fit follow-up).
- Define 2 new ledger action types.

Cost: ~3h. Risk: low (pattern known; both affected skills already ceremonial). Value: enables CEO-emitted invariant announcements (e.g., "v5.23 live") without spamming N peer-to-peer memos.

### Phase B — infrastructure (target: days 3–4)

**B1. Committee 02 — approval chains (minimal extension)**

- Create `references/approval-chains.toml` with 3 chain definitions.
- Ship `bin/approval_chain.py` helper.
- Refactor `bin/hooks/pre_write_agent.py` + `pre_write_kb.py` to be chain-agnostic.
- Extend `bin/ceo_behavior_audit.py` with `approval_chain_skipped` drift class.
- Add 4 new ledger action types.
- Update v5.22 replay harness with approval-chain scenarios.

Cost: ~4–6h. Risk: MEDIUM — refactoring shipped v5.22 hooks introduces regression surface. Must run replay harness before landing. Value: makes adding chains (okr-individual, future) a ≤ 20-line TOML change with no Python edits.

**B2. Committee 06 — dashboard (regenerable T2)**

- Ship `bin/dashboard.py` with 6 of 7 metrics (metric 7 stubbed pending committee 05).
- Edit `skills/core/ceremony/retrospective/SKILL.md` to load dashboard at retro open.
- Add `dashboard-period-md` data-storage-matrix row (Storage-fit follow-up).

Cost: ~6h. Risk: low (script-only, regenerable). Value: retrospective ceremonies gain quantitative anchor; committees 01 + 05 become observable.

### Phase C — agent-visible features (target: days 5–6)

**C1. Committee 05 — 360 review (cycle-outcome hybrid)**

- Ship `bin/agent_cycle_score.py` with per-agent score formula + unit tests.
- Edit `skills/core/hr/agent-promote/SKILL.md` to cite the score in promotion criteria.
- Add `agent-cycle-score-jsonl` data-storage-matrix row (Storage-fit follow-up).
- Backfill dashboard.py metric 7 to pull from this source.

Cost: ~4h. Risk: MEDIUM — scoring formula is uncalibrated; Goodhart risk mitigated by values-alignment-audit (existing). Explicit user-flag note in `decision.md` acknowledges the speculative weighting.

**C2. Committee 01 — OKR framework (3 new skills)**

- Author 3 skills (`okr-set`, `okr-checkin`, `okr-close`) via skill-factory 10-step pipeline.
- Register `okrs/<period>/<o-id>.md` T1 row via Storage-fit committee.
- Wire `okr-set` RACI pre-emit gate — mechanism via committee 02's approval-chain DSL (now landed in phase B1).
- CATALOG.md updates.

Cost: ~6h. Risk: MEDIUM — skill-factory authoring of 3 interrelated skills requires careful Phase 2 subagent-request batching. Value: closes the user's headline gap; OKRs expressible + committee-reviewable.
