## Round 1

### research

- **@kiho-eng-lead** (confidence: 0.85) — Existing regenerable-T2 precedent: `cycles/INDEX.md` regenerated at CEO DONE step 11, `.kiho/state/org-registry.md` regenerated by `org-sync`, `skills/CATALOG.md` semi-manually maintained. Pattern is deterministic-script-reads-sources-emits-markdown. Dashboard fits this cleanly; no new architectural precedent needed.
- **@kiho-kb-manager** (confidence: 0.80) — If T2 regenerable, the dashboard never needs a KB_MANAGER_CERTIFICATE and doesn't route through me. Freed from gateway duty. Alternative: T1 synthesized via me would add narrative (e.g., "this period saw a hiring surge because X"), but narrative cost vs marginal-insight value is low.
- **@kiho-perf-reviewer** (confidence: 0.83) — Telemetry sources ready to feed a dashboard: `cycle-events.jsonl`, `agent-performance.jsonl`, `factory-verdicts.jsonl`, `skill-invocations.jsonl`, committee-close outcomes, and the incoming `agent-score-<period>.jsonl` from committee 05. Together these cover velocity, hiring, incidents, skill churn, committee activity, and the new score rollup.
- **@kiho-comms** (confidence: 0.82) — Retrospective ceremony is the natural consumer. Retro today is narrative-only; adding a quantitative preface that the script generates would anchor the conversation in numbers.
- **@kiho-auditor-cost-hawk** (confidence: 0.72) — Every metric we include must earn its place. If no downstream consumer references a metric, it's decoration. Propose: metric list starts at ZERO; each addition requires a named consumer during this committee. No "just in case" metrics.

### suggest

- **@kiho-eng-lead** (confidence: 0.85) — Metric list, each with consumer:

| Metric | Source JSONL | Consumer |
|---|---|---|
| cycles_closed / opened this period | cycle-events.jsonl | retrospective (velocity) |
| incidents_opened + mean MTTR | cycle-events.jsonl (incident-lifecycle entries) | retrospective (reliability) |
| agents_hired + agents_rejected | ledger entries `recruit_*` + rejection-feedback outcomes | HR ceremony / retrospective |
| committees_convened + unanimous_close_rate | committee transcript closes | retrospective + skill-evolution |
| skills_factory_pass_rate + reject_rate | factory-verdicts.jsonl | skill-evolution committee |
| kb_pages_added | kb-add entries in ledger | kb-lint health check |
| top_5_agent_scores + bottom_5 | agent-score-<period>.jsonl (committee 05 output) | agent-promote committee |

Seven metrics. Each with explicit consumer.

- **@kiho-eng-lead** (confidence: 0.85) — Shape: `bin/dashboard.py` stdlib script invoked by retrospective ceremony. Output: `.kiho/state/dashboards/<period>.md` T2 regenerable. No new skill.

### challenge

- **@kiho-comms** (confidence: 0.83) — Challenge on cadence: period=? Cost-hawk wants "earn its place"; does WEEKLY earn its place over PER-CYCLE or QUARTERLY?
  > Resolved: period is PER-CYCLE + QUARTERLY. Per-cycle because retrospective ceremony fires at cycle close; quarterly because committee 01 (OKR) operates on quarters + agent-score rollup is quarterly. Daily/weekly rejected — fires too often without natural ceremony anchor.
- **@kiho-auditor-cost-hawk** (confidence: 0.78) — Challenge on `top_5_agent_scores` metric: is it actionable or voyeuristic?
  > Resolved: consumer is `agent-promote` committee per committee 05's decision. Top 5 identifies promotion candidates; bottom 5 identifies performance-review priority. Actionable. Metric earns its place.
- **@kiho-kb-manager** (confidence: 0.82) — Challenge on `kb_pages_added` metric: consumer is "kb-lint health check" but kb-lint runs on every KB write, not on dashboard read. Is this metric redundant?
  > Resolved: kb-lint checks individual-page health; the dashboard metric tracks RATE (growth / stagnation over period). Different question, different consumer. Retain.

### choose

- **@kiho-kb-manager** (confidence: 0.90) — T2 regenerable, 7 metrics, per-cycle + quarterly cadence, new `bin/dashboard.py` script invoked from retrospective.
- **@kiho-comms** (confidence: 0.92) — Same.
- **@kiho-perf-reviewer** (confidence: 0.90) — Same.
- **@kiho-eng-lead** (confidence: 0.92) — Same.
- **@kiho-auditor-cost-hawk** (confidence: 0.90) — Same. All 7 metrics have named consumers, threshold earned.

Aggregate = mean(0.90, 0.92, 0.90, 0.92, 0.90) = **0.908** ≥ 0.90. Unanimous. No unresolved challenges. **CLOSE.**

## Close

ZERO new skills; one new Python script; one new T2 data class."
