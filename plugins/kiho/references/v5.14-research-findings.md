## Headline findings

### H1. Isolation + grader-correctness review, not more graders

Anthropic's Jan 9 2026 "Demystifying Evals for AI Agents" post establishes two non-negotiable eval principles:

1. **Environment isolation per trial** — "each trial should be isolated by starting from a clean environment" to prevent skill A leaking state into skill B's eval.
2. **Mandatory transcript review of failures AND passes** — "you won't know if your graders are working well unless you read the transcripts and grades from many trials."

The post also distinguishes **capability evals** (low pass rate, "hill to climb") from **regression evals** (~100% pass rate, "cliff to avoid falling off") and treats them as different artifacts with different lifecycles. v5.13 conflated them.

**v5.14 integration:**
- Split eval buckets into `evals/capability/` and `evals/regression/` (reference: `skills/_meta/skill-create/references/capability-regression-split.md`)
- Add Gate 12 isolation manifest (`scripts/isolation_manifest.py`)
- Add Gate 13 grader review (`scripts/grader_review.py`) with 10% deterministic sampling
- Switch description scoring from raw accuracy to precision + recall + F1 + balanced_accuracy (`scripts/compute_precision_recall.py`)

### H2. Analyzer + comparator pair with assertion-discrimination scoring

The `anthropics/skills/skill-creator` commit `b0cbd3d` (Mar 6 2026) introduced three artifacts v5.13 did not have:

1. **`agents/analyzer.md`** — examines benchmark.json for non-discriminating assertions, flaky evals, and time/token tradeoffs
2. **`agents/comparator.md`** — blind A/B between iterations, picks a winner via 4-dim rubric
3. **Two new schemas** — `analysis.json` (`improvement_suggestions`, `transcript_insights`) and `comparison.json` (`winner_strengths`, `loser_weaknesses`, `instruction_following_delta`)

**Novel primitive: assertion-discrimination delta.** For every assertion in the grading rubric, compute `pass_rate_with_skill - pass_rate_without_skill`. Assertions with delta < 0.20 are weakly discriminating; delta < 0.00 means the skill made things WORSE. A rubric where >50% of assertions don't discriminate has zero information value.

**v5.14 integration:**
- New sub-agents `agents/analyzer.md` and `agents/comparator.md` under `skills/_meta/skill-create/`
- New script `scripts/compute_discrimination.py`
- New reference `references/analyzer-comparator.md`
- New Step 10.5 in skill-create pipeline (after Gate 11 transcript review, before Step 11 registration)
- Non-monotonic iteration rule: the winning iteration is NOT necessarily the most recent one (per Anthropic's Mar 24 2026 harness-design post: "I regularly saw cases where I preferred a middle iteration over the last one")

### H3. Hierarchical catalogs beat flat at 200+ skills

**AgentSkillOS** (arXiv 2603.02176, March 2026) measures Bradley-Terry scores across 200 → 200K skills. At 200 skills, flat skill provision scores 24.3 vs AgentSkillOS hierarchical 100.0. At 200K skills, flat collapses to 17.2 while hierarchical retrieval holds at 100.0. This is **causal** evidence (oracle-set comparison) that the catalog architecture matters, not just retrieval quality.

Claude Code's own budget math reinforces this:
- `SLASH_COMMAND_TOOL_CHAR_BUDGET` defaults to **1% of context window** with **8,000-char fallback**
- Each skill's combined `description + when_to_use` is **hard-capped at 1,536 chars** regardless of total budget
- A flat catalog of 200 skills would saturate the budget and start dropping descriptions silently

**v5.14 integration:**
- Promote `skills/{_meta,core,kb,memory,engineering}/` from convention to structural contract
- Add per-domain `routing-description` in `skills/CATALOG.md` (the new YAML-fenced `routing:` block at the top)
- New Gate 14 catalog-fit (`scripts/catalog_fit.py`): new skill description must overlap parent domain's routing-description by ≥1 substantive keyword
- New Gate 15 budget pre-flight (`scripts/budget_preflight.py`): total ACTIVE description chars ≤ 90% of budget; per-skill ≤ 1,536 chars
- New Gate 16 compaction budget (`scripts/compaction_budget.py`): top-N recent-invocation skills ≤ 80% of 25k post-compaction ceiling
- Plan: depth-cap escalation when catalog crosses ~80 skills

### H4. Security: Gate 9 is a 2025-vintage defense against a 2026 attack class

Four primary sources replaced the "Lethal Trifecta" heuristic with hard numbers and structured taxonomies:

- **Snyk ToxicSkills** (Feb 5 2026) — 3,984 skills scanned, 13.4% critical, 36.82% any flaw, 76 confirmed malicious, 8-category taxonomy, `mcp-scan` open-source tool with 90-100% recall
- **arXiv 2602.12430** (Feb 2026) — 26.1% of 31,132 community skills have vulnerabilities, scripts increase risk 2.12×, proposes 4-gate + 4-tier framework with runtime promotion/demotion
- **arXiv 2604.02837** (March 2026) — 7-category threat taxonomy, explicitly concludes static-AST malicious intent detection is infeasible, pivots to delta-consent + version-bound trust
- **arXiv 2510.26328** (Oct 2025) — demonstrates permission escalation via skill files ("don't ask again")

**Critical negative result**: three independent groups all concluded that **AST-based malicious intent detection is fundamentally infeasible** because natural-language SKILL.md collapses the data/instructions boundary. Don't chase this.

**v5.14 integration:**
- Replace Lethal Trifecta check with Snyk 8-category taxonomy (reference: `skills/_meta/skill-create/references/security-v5.14.md`)
- Add `metadata.trust-tier: T1|T2|T3|T4` frontmatter field; CEO-only promotion past T2
- Apply **2.12× rule**: script-bearing skills get 1 extra eval, 1 extra transcript review scenario, tighter token budget, and stricter delta-consent thresholds
- Add **delta-consent**: when an active skill changes by >10% bytes (or ANY script change for script-bearing skills), downgrade to T1 and require re-promotion through the committee
- Update Gate 9 procedure to run the 8-category sweep + trust-tier check + script-scan flag

### H5. Evaluator-generator separation; non-monotonic iteration

Anthropic's Mar 24 2026 "Harness design for long-running application development" reports that when Anthropic asked agents to evaluate work they had produced, they "tend to respond by confidently praising the work — even when quality is obviously mediocre." Their fix: a **separate evaluator agent** with skeptical-by-default tuning and concrete criteria.

> "Tuning a standalone evaluator to be skeptical turns out to be far more tractable than making a generator critical of its own work."

They also note **iterations are not monotonic**: "I regularly saw cases where I preferred a middle iteration over the last one." This means comparing N vs N-1 is wrong — you must compare N against ALL previous iterations and pick the historical best.

**v5.14 integration:**
- Gate 11 transcript review now uses a **fresh skeptical evaluator subagent**, never the agent that authored the skill and never the agent that ran the scenario
- System prompt prefix: "uncertainty defaults to FAIL; praise is affirmative and must be earned"
- `scripts/run_loop.py` compares each new iteration against the current best (not the immediately-previous), and `run-loop.json` reports `non_monotonic_winner: true` when the historical best beats the most recent iteration
- The winning iteration's SKILL.md is copied to `sk-<slug>/SKILL.md`, not the newest file on disk

## Per-thread findings

See the research report for full per-thread breakdowns. Key smaller findings absorbed into v5.14:

**Thread 1 — `anthropics/skills` source:** precision + recall + accuracy (not just accuracy), stratified train/test split balanced by should_trigger, "generalize away from failed queries" prompting (from Mar 6 2026 commit `b0cbd3d`), grader `claims[]` field with uncertainty-defaults-to-FAIL rule.

**Thread 2 — Claude Code docs:** new `when_to_use` frontmatter field (appended to description, counts toward 1,536-char cap), skill-scoped `hooks`, `shell: powershell`, `disableSkillShellExecution` setting, post-compaction 25k-token ceiling, subagent skill preloading (depth-2 subagents with `skills:` list eat full skill content at startup, not lazy).

**Thread 3 — agentskills.io spec:** 6 canonical frontmatter fields confirmed (`name`, `description`, `license`, `compatibility`, `metadata`, `allowed-tools`); `compatibility` is capped at 500 chars; kiho extensions must live under `metadata:`; `skills-ref validate` CLI available for defense-in-depth schema check.

**Thread 5 — Community ecosystem:** Anthropic explicitly declined the declarative-dependency proposal (`anthropics/claude-code#27113`, Feb 20 2026, closed not-planned). kiho must own its own `requires:` chain forever. Embedding-based skill retrieval exists in the wild (`K-Dense-AI/claude-skills-mcp`) but only as an external wrapper, never inside Claude Code itself.

**Thread 7 — Eval methodology:** pass@k vs pass^k distinction (skills for user-visible artifacts need pass^k; skills for option-proposing need pass@k only). Add per-skill `eval-mode` declaration. Re-run all gates on new model upgrades (Anthropic removed sprint contracts on Opus 4.5 → 4.6 upgrade).

**Thread 8 — Composition and orchestration:** AgentSkillOS DAG strategies — Quality-First (adds refinement), Efficiency-First (parallelizes), Simplicity-First (minimal graph). kiho's CEO loop maps to Simplicity-First because of depth cap 3 and fanout cap 5. The "top-level orchestration skill" community pattern (not embedded sequencing) aligns with kiho's `_meta/` directory.

**Thread 10 — Multi-agent skill sharing:** literature is thin. Anthropic's harness blog implies role-scoped skill portfolios (planner/generator/evaluator each has its own), not globally shared pools. Recommend adding `intended-roles: [ceo, dept-lead, ic]` metadata field to skills and a fragmentation check during promotion.

## What was deliberately NOT added

These findings were considered and rejected with reasons:

1. **AST-based malicious intent detection** — three independent papers say infeasible
2. **Embedding-based skill retrieval inside kiho** — violates markdown+JSON only constraint; Claude Code itself doesn't do it
3. **`dependencies.required[]` field** mirror of rejected Claude Code proposal #27113 — would fragment standard and create maintenance debt
4. **Cross-model consensus judging at Gate 11** — too expensive in single-turn budget; stick to single-judge with evaluator separation
5. **Auto-promotion past trust tier T2** — CEO-only decision; reintroducing the ClawHavoc pattern otherwise
6. **`mcp-scan` as a hard blocking gate** — advisory only; community tool at T2 trust; release cadence coupling
7. **`when_to_use` required globally** — pushes catalog toward silent truncation; only required when description < 200 chars
8. **pass^k for every skill** — use per-skill `eval-mode` field; many kiho skills (skill-learn candidates) only need pass@k
9. **Replace 11 gates with G1-G4 security framework** — different concerns; ADD G2 + G4 as new Gates 12-13, keep the existing 11
