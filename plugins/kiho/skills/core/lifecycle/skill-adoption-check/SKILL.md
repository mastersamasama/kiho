# skill-adoption-check

The "did anyone use it?" follow-up. Without this skill, the org ships and forgets. With it, every new skill gets an automatic adoption review at 30/60/90 days; zero-adoption skills get FIXed before the description-trigger gap calcifies.

> **v5.21 cycle-aware.** This skill is the `adoption-monitor` phase entry in `references/cycle-templates/talent-acquisition.toml` (90-day post-onboard scope=agent variant) and reachable as a future `monitor` phase in `skill-evolution.toml`. When run from cycle-runner, scope is narrowed to a specific agent or skill via the cycle's `index.params`; this skill writes first_check_done back into `index.adoption.*`. Atomic invocation (org-wide monthly cron) remains the primary path.

## When to use

- Auto-fires monthly from CEO INITIALIZE (default cadence)
- Manually after a skill-factory batch ships ≥ 5 new skills (verify the batch landed)
- Pre-evolve agenda assembly: skill-adoption-check is the source of "skill ships, nobody uses, queue for FIX"

Do **NOT** invoke when:

- The skill is < 14 days old — too early; let real workflows discover it
- The skill is `lifecycle: deferred` — it's not supposed to be used yet
- The skill is `lifecycle: deprecated` — zero adoption is expected; check skill-deprecate's consumer migration log instead

## Inputs

```
window_days: <int>              # default 90 (covers 30/60/90 markers)
markers: [<int>, ...]           # default [30, 60, 90]
zero_threshold: <int>           # default 0 (literal: zero invocations)
include_low_adoption: <bool>    # default true; flag <5 invocations as low-adoption-warning
output_to_evolve_agenda: <bool> # default true
```

## Procedure

1. **Enumerate candidate skills.** Walk `skills/**/SKILL.md`. Filter to those where:
   - `lifecycle: active`
   - `created_at` from frontmatter is within `window_days` from today

2. **For each candidate, count invocations** from `<project>/.kiho/state/skill-invocations.jsonl` filtered by `skill_id` AND `ts >= created_at`.

3. **For each marker (30/60/90 days), compute adoption status:**
   - **zero-adoption**: count == 0 at this marker
   - **low-adoption**: count < 5 at the 60+ marker (warning, not flag)
   - **adopted**: count >= 5 at the 30 marker, or count >= 10 at the 60 marker

4. **Build the adoption report.** For each candidate, capture:
   - skill_id, created_at, age_days
   - per-marker counts, status
   - top 3 invoking agents (if any)
   - dominant success/failure ratio (from skill-health.jsonl row if exists)
   - hypothesis on why zero (auto-generated): `description-trigger-mismatch | wrong-domain | covers-problem-nobody-has | duplicate-of-existing`

5. **Write the adoption report.** Call `storage-broker` op=`put`:
   ```
   namespace: state/adoption-checks/<check_id>
   kind: adoption-check
   access_pattern: read-mostly
   durability: project
   human_legible: true
   body: { check_id, ts, candidates_count, per_skill_results, flagged_zero_adoption,
           flagged_low_adoption, evolve_agenda_appended }
   ```

6. **Append to evolve agenda** (if `output_to_evolve_agenda: true`). For each zero-adoption skill at the 30+ marker:
   ```
   Append to <project>/.kiho/state/evolve-agenda.jsonl:
   { ts, skill_id, trigger: "zero-adoption-at-day-<marker>",
     hypothesis: "<auto-generated>", recommended_op: "skill-improve",
     evidence_ref: "<adoption_report_ref>" }
   ```
   The next `/kiho evolve` (manual or scheduled) reads this file and runs FIX iterations.

7. **CEO notification** via single `memo-send severity=fyi` with the report ref. Severity `action` if any zero-adoption skills are present.

8. **Memory observations to skill-authors** (type=observation, importance=4, written_by=ceo-01): "Skill <id> you authored on <date>: zero adoption at day-30; queued for FIX with hypothesis <h>". This loops the author into the signal so future authoring choices learn from it.

## Invariants

- **Marker, not threshold.** A skill is "zero-adoption at day-30", not "zero-adoption ever". Time matters.
- **Zero-adoption is a hypothesis-generation event, not a deletion event.** Never auto-deprecate from this skill; that's the author + committee call after FIX attempts.
- **Hypothesis is a guess, not a verdict.** The auto-generated hypothesis biases the FIX direction; skill-improve still does its own diagnosis.
- **Author always notified.** Even if zero-adoption is later explained (real adoption was via a derived skill, etc.), the author should know.

## Non-Goals

- Not a skill-deprecation tool. Use `skill-deprecate` for that.
- Not a skill-quality scorer. That's `skill-critic`.
- Not a usage analytics dashboard. The report is operational triage, not org-wide reporting.
- Not for measuring CEO skills (sk-001 etc.). Those are loaded into every CEO turn; "adoption" doesn't apply.

## Anti-patterns

- Never run with `zero_threshold > 0`. The point is to catch genuine non-adoption.
- Never silence the author notification. Hiding signal from authors prevents the org from learning to write better trigger descriptions.
- Never re-add a previously-flagged skill to the evolve agenda within the same window. One FIX iteration per check; if the FIX didn't help, that's a reposition / deprecate decision, not a re-flag.
- Never check skills < 14 days old. Adoption noise dominates real signal in the first two weeks.
