# soul-apply-override

The soul-evolution commit step. Takes a bundle of reviewed override entries and merges them into an agent's canonical `.md` soul document, then broadcasts the new state to downstream systems. This is the only skill authorized to mutate an agent's Soul section outside of the initial recruit flow.

## Inputs

```
agent_id: <agent name, e.g., "eng-lead-01">
override_entries: <optional — list of override entry IDs to apply; defaults to all pending for this agent>
authorized_by: <ceo | hr-lead — who approved the overrides>
dry_run: <bool — if true, show the merge result without writing>
```

Pending overrides live in `.kiho/agents/<agent_id>/soul-overrides.md` (queue) and are produced by `memory-reflect`.

## Procedure

1. **Auth check** — confirm `authorized_by` has the right to apply these overrides. Red-line section edits require `ceo` AND a resolved `values_flag_ref` (ruling_id, e.g. `vf-2026-04-19-0001`) pointing to a `values-flag` artifact at `.kiho/state/values-conflicts/<ruling_id>.md` with `ceo_signed` non-empty and `soul_mutation_required: true`. Missing or unresolved ref → `status: error` with detail `"values-flag ruling required for red-line override"`; no CEO bypass. All non-red-line sections may be applied by `hr-lead`.
2. **Load current state** — read `agents/<agent_id>.md` and the pending queue. Capture the current `soul_version` from frontmatter.
3. **Validate overrides** — each entry must specify `section`, `operation` (add/replace/remove/rerank), `payload`, `rationale`. Reject malformed entries with `status: error`.
4. **Apply merge** — walk each entry and compute the new Soul section content per the [merging rules](#override-merging-rules). Do not touch sections outside Soul.
5. **Coherence recheck** — run the soul-architecture coherence checklist: Big Five values still in [1,5], voice line still non-empty, values count 3-7, goals count 1-5, red lines non-empty.
6. **Bump version** — increment `soul_version` (see [Versioning](#versioning)).
7. **Append trait history** — add an entry to `.kiho/agents/<agent_id>/trait-history.jsonl` per applied override (old value, new value, rationale, timestamp, authorized_by).
8. **Write agent .md** — persist the updated soul section back to `agents/<agent_id>.md`. If `dry_run`, skip this and return the diff.
9. **Mark overrides applied** — move consumed entries from the pending queue to `.kiho/agents/<agent_id>/soul-overrides-applied.md`.
10. **Propagate** — call `org-sync` (refresh org-registry.md), `kb-update` (update the entity page for this agent), and `memory-cross-agent-learn` (broadcast a lesson to same-team peers summarizing the drift).

## Override merging rules

- **Additive sections** (values, goals, red lines) — `add` appends, `remove` drops by string match, `replace` swaps one entry for another.
- **Numeric sections** (Big Five traits) — `replace` is the only allowed operation. New value must stay in [1,5]. Deltas > 1.0 in a single override require `ceo` auth.
- **Ranked sections** (values list is ordered by priority) — `rerank` accepts a new ordering; the set of items must be identical.
- **Red lines** — any edit (add, remove, replace) requires `ceo` auth AND a resolved `values_flag_ref` per Step 1. Never applied by hr-lead; never applied without a values-flag ruling.
- **Immutable sections** — `name`, `tier`, `created_at`, `parent_agent`, and the voice-anchor examples in `references/soul-architecture.md` may NEVER be edited via override. Reject with `status: error`.

## Section edits format

Each override entry in the queue uses this YAML shape:

```yaml
- entry_id: ov-2026-04-14-001
  section: values
  operation: add
  payload: "transparency over cleverness"
  rationale: "Observed 3 instances where agent chose clarity over terseness in review feedback."
  authorized_by: ceo
  created_at: 2026-04-13T22:05:00Z
```

The skill processes entries in `created_at` order. Conflicts (two entries touching the same section) resolve last-write-wins with a warning in the response.

## Versioning

Soul versions use `MAJOR.MINOR` semver-lite:

- **Patch / minor merge** (single additive change, no red-line touch) — bump MINOR: `v5` → `v5.1` → `v5.2`
- **Major merge** (red-line edit OR > 3 entries in one batch OR Big Five delta > 1.0) — bump MAJOR: `v5.2` → `v6`

Version history lives in `.kiho/agents/<agent_id>/trait-history.jsonl` and is never rewritten.

## Response shape

```json
{
  "status": "ok | error | dry_run",
  "agent_id": "eng-lead-01",
  "soul_version_before": "v5.1",
  "soul_version_after": "v5.2",
  "applied_overrides": ["ov-2026-04-14-001", "ov-2026-04-14-002"],
  "before_after_traits": {
    "values": {"before": ["rigor", "speed"], "after": ["rigor", "speed", "transparency over cleverness"]}
  },
  "coherence_recheck": "ok",
  "trait_history_entries_added": 2,
  "propagation": {
    "org_sync": "ok",
    "kb_update": "ok",
    "memory_cross_agent_learn": "queued"
  }
}
```

## Anti-patterns

- **Skipping the coherence recheck.** A merge that lands an out-of-range Big Five value or empties the voice line must be rolled back before write.
- **Editing immutable sections.** Name, tier, and created_at are part of the agent's identity — treat them as write-once.
- **Silent rollback.** If validation fails mid-merge, restore the original .md and return `status: error` with the failing entry. Never leave a half-merged soul on disk.
- **Bypassing the pending-queue rate limit.** At most 3 overrides may be applied per turn; additional entries wait for the next `/kiho evolve` cycle.
- **Authorizing red-line changes as hr-lead.** Red lines define the agent's ethical contract; only the CEO may touch them.
- **Applying a red-line override without a resolved values-flag ruling.** v5.20+ requires every red-line mutation to cite a `values_flag_ref` pointing to a committee-reviewed, CEO-signed `values-flag` artifact. CEO sign-off alone is insufficient — the ruling provides the auditable provenance chain back to the soul clause that was challenged.
