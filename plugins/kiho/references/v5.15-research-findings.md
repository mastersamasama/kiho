## Methodology

An isolated general-purpose research agent was spawned on Apr 15 2026 with a clean kiho brief (no codebase context). The agent was told: "prefer primary sources (Anthropic docs, `anthropics/skills` repo, arXiv 2024-2026, engineering blogs from Anthropic/OpenAI/DeepMind, major skill ecosystems, package-manager docs); cite every finding with a real URL; return a structured report with headline findings, per-question answers, and a rejected-recommendations list." Budget: 20-30 WebSearches, 15-25 WebFetch calls, ~40 minutes wall clock.

The brief explicitly stated kiho's constraints: markdown + JSON state only, no runtime database, no vector index, no MCP server, depth cap 3, fanout cap 5. Recommendations that violated those constraints were marked "does not fit" in the report and excluded from v5.15 scope.

## Headline findings

### H1. Upstream skill-creator has no pre-create similarity check — kiho is building greenfield

Reading `github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md` directly: there is no duplicate detection, no overlap check, no "improve existing vs create new" branch. The v2.0 update (Mar 2026) added Create/Eval/Improve/Benchmark modes, but the "Improve" mode assumes the skill has already been identified — it does not scan the catalog for existing overlaps. The closest thing to a similarity check is a suggestion to "research in parallel via subagents" during the Interview phase, which is not a mechanical gate.

**Implication for kiho:** adding a similarity check is *greenfield territory*, not cowpath-paving. The design cost should be deliberately justified. It is, but the justification comes from H3, not from upstream precedent.

**v5.15 integration:** Gate 17 + `similarity_scan.py` + `novel-contribution.md` reference. The design is explicitly documented as a kiho extension beyond upstream.

Source: https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md

### H2. Declarative skill dependencies were explicitly rejected upstream

Claude Code issue #27113 proposed a `dependencies.required` / `dependencies.recommended` block in `.claude/settings.json`. It was closed "as not planned" with no maintainer reasoning on record — just a stale label. A sister issue #9444 (plugin-to-plugin dependencies for shared agents) is still open with no decision. Further: agentskills RFC #252 proposed a `signature` frontmatter field and was rejected by maintainer Jonathan Hefner on the basis that "signatures are better implemented at the distribution layer" — establishing precedent that **structural metadata belongs outside the skill file when possible**.

**Implication for kiho:** kiho can add declarative dependencies locally, but cannot expect upstream spec support and must not use frontmatter fields that might collide with future upstream additions. Everything must live under `metadata.kiho.*` to stay spec-compliant.

**v5.15 integration:** the five dep fields (`requires`, `mentions`, `reads`, `supersedes`, `deprecated`, `superseded-by`) are all namespaced under `metadata.kiho.*`. Top-level `requires:` is explicitly rejected by Gate 2 as a spec violation.

Sources:
- https://github.com/anthropics/claude-code/issues/27113
- https://github.com/anthropics/claude-code/issues/9444
- https://github.com/agentskills/agentskills/discussions/252

### H3. Selection accuracy collapses as a phase transition driven by semantic confusability, not size

arXiv 2601.04748 ("When Single-Agent with Skills Replace Multi-Agent Systems…") directly measures the skill-selection accuracy curve:

> **§5.2:** "At small scales (|S| ≤ 20), accuracy remains above 90%, but degrades steadily beyond |S| = 30, falling to approximately 20% at |S| = 200."
>
> **§5.3:** "At identical |S| = 20, replacing unique skills with base-competitor pairs causes an 18–30% accuracy drop. This demonstrates that semantic structure determines selection difficulty."

Their fitted scaling law is `Acc(σ, S) ≈ α / (1 + (|S|/κ)^γ) − ε·I(S)`, where κ is the capacity threshold and ε is sensitivity to semantic interference `I(S)`. *Two similar skills hurt more than two extra unrelated skills.*

**Implication for kiho:** kiho at 37 registered skills is already past the |S| = 30 inflection. The right metric to budget against is **mean pairwise Jaccard** (or any equivalent confusability proxy), not skill count. The enemy is overlap, not growth. Gate 17's threshold-based block is the mechanism.

**v5.15 integration:** Gate 17 uses Jaccard ≥ 0.60 as the block threshold, citing arXiv 2411.04257 (LSHBloom) and Nelhage fuzzy-dedup for the specific number. The `--catalog-health` mode of `similarity_scan.py` reports mean-pairwise Jaccard as an early-warning metric; the current April 2026 baseline for kiho's 37-skill catalog is 0.015 (very clean).

Source: https://arxiv.org/html/2601.04748v1

### H4. Mechanical skill merging is an open problem in the agent-skill literature

AgentSkillOS (arXiv 2603.02176 §2.1.1) is the only paper surveyed that has a mechanical merge rule at all, and it merges *categories with a single child*, not skills themselves:

> "If the number of skills in any generated category after Skill Assignment is equal to 1, that category and all its skills will be merged into the most relevant target category."

That is category merge, not skill merge. Merging two overlapping **skills** into one — preserving the strengths of each — is not addressed anywhere in the 2024–2026 skill literature. arXiv 2602.12430 (the skill survey) acknowledges the problem in its Challenge 2 section but proposes no procedure. Git has no corresponding primitive either: `ort`/`recursive` auto-resolve where diffs don't conflict, and `-X ours`/`theirs` resolve conflicts by picking one side — neither is "preserve both strengths." Every merge-preserving operation in git is a human writing a new third artifact from diff inputs.

**Implication for kiho:** do **not** build a mechanical "merge two skills" primitive. That path has no prior art and would almost certainly fail. Provide mechanical *overlap inputs* (Gate 17's JSON with shared tokens + unique-to-each samples) and let humans write the merged third artifact — either via `skill-improve` on the existing skill (absorbing good ideas from the draft) or via a fresh third skill with a distinctive description.

**v5.15 integration:** Gate 17's `suggested_action` is `improve <top-match>`, never `merge`. The failure route documented in `novel-contribution.md` is explicit: use `skill-improve`, `skill-derive`, or re-specify; do not attempt a merge.

Sources:
- https://arxiv.org/html/2603.02176v1 (§2.1.1)
- https://arxiv.org/html/2602.12430 (§4.6, §7)
- https://git-scm.com/docs/merge-strategies

### H5. Reverse-dependency tracking in every mature ecosystem is computed on demand, never stored

Every mature package manager surveyed walks forward edges at query time:

- **`pnpm why <pkg>`** walks the forward lockfile in reverse with subtree deduplication (pnpm 10.30 redesign). No reverse index on disk.
- **`cargo tree -i <crate>`** inverts the resolved Cargo graph at query time. `-e features` extends to feature flow.
- **`go mod why <pkg>`** answers "does main depend on X" via the module graph. Notoriously lossy for indirect use.
- **`bazel query 'rdeps(u, x)'`** operates within a universe scope; Sky Query mode introspects the Skyframe graph directly. "Precomputed" only because Skyframe is kept resident in server memory — not persisted to disk.
- **Kubernetes `ownerReferences`** is the one runtime exception: each dependent object forward-points to its owner, and the garbage collector builds a reverse index at runtime by watching the API. Still in-memory, still runtime-only, never persisted.
- **Terraform** stores a forward DAG and walks it in reverse order for `terraform destroy`. No stored reverse adjacency.

The universal pattern: **forward edges are the source of truth; reverse views are computed, sometimes cached in process memory, never persisted to disk as a separate artifact**. The reason is always the same: a persisted reverse index is almost always stale at the moment of query.

**Implication for kiho:** do **not** store reverse dependencies on disk. Store forward refs only (in SKILL.md frontmatter, under `metadata.kiho.*`) and compute reverse refs at query time by scanning forward edges across the tree.

**v5.15 integration:** `bin/kiho_rdeps.py` is a pure forward-edge scanner with zero on-disk cache. Each invocation walks the tree fresh. At 37 skills the walk completes in <500ms; a future in-memory cache is acceptable but a disk cache is not.

Sources:
- https://pnpm.io/cli/why
- https://doc.rust-lang.org/cargo/commands/cargo-tree.html
- https://go.dev/ref/mod
- https://bazel.build/query/language
- https://kube.rs/controllers/relations/
- https://developer.hashicorp.com/terraform/internals/graph

## Per-question answers (Q1–Q10)

**Q1. Pre-create similarity check in `anthropics/skills/skill-creator` and forks.** Does not exist. The v2.0 update added Create/Eval/Improve/Benchmark but "Improve" operates on an already-identified skill. K-Dense-AI/claude-skills-mcp provides semantic skill *discovery* via sentence-transformers vector search, but that's a runtime lookup tool running as an MCP daemon — not a pre-create gate. VoltAgent/awesome-agent-skills and alirezarezvani/claude-skills are aggregators with no overlap detection. Sources: https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md, https://github.com/K-Dense-AI/claude-skills-mcp.

**Q2. Embedding vs keyword similarity in markdown-only ecosystems.** K-Dense-AI/claude-skills-mcp uses sentence-transformers (specific model undisclosed) to build a runtime index of ~90 skills — it's an MCP server because it needs a daemon. For kiho's markdown + no-runtime-index constraint, embedding search is inappropriate: it needs a daemon or a rebuild step whose stale index silently misleads you. MinHash + Jaccard on description shingles is the 2024-2025 industry standard for markdown-scale dedup (LSHBloom, Milvus MinHash LSH); deterministic, explainable, zero runtime deps. For kiho's 37-skill scale, direct pairwise Jaccard is even simpler than MinHash. Sources: https://blog.nelhage.com/post/fuzzy-dedup/, https://arxiv.org/html/2411.04257, https://github.com/K-Dense-AI/claude-skills-mcp.

**Q3. Reverse-dep in package managers.** Answered in H5. Universal pattern: walk forward edges at query time. `pnpm why`, `cargo tree -i`, `go mod why`, `bazel rdeps`. Rename/deprecate uses a shim pattern: old package publishes a thin re-export wrapper around new package; consumers discover via build-time warnings. Sources: https://pnpm.io/cli/why, https://doc.rust-lang.org/cargo/commands/cargo-tree.html.

**Q4. Mechanical skill merging.** Not in the literature. Answered in H4. arXiv 2602.12430 §4.6 discusses "Skill Compilation" but that's converting multi-agent systems into skill libraries, not merging overlapping skills. AgentSkillOS merges categories with a single child. Nothing about merging two skills into one. **Open problem as of April 2026.** Source: https://arxiv.org/html/2602.12430.

**Q5. agentskills.io spec fields.** Six canonical fields: `name` (required, ≤64 chars), `description` (required, ≤1024 chars), `license`, `compatibility` (≤500 chars), `metadata` (free-form map), `allowed-tools` (experimental). **No `requires`, no `depends-on`, no `extends`**. `allowed-tools` is a permission list, not a dependency list, and issue #18837 reports it is not actually enforced by the Claude Code harness. RFC #252 (signature field) rejected on "structural metadata belongs outside the skill file" precedent. Issue #27113 (declarative deps) closed "not planned". Sources: https://agentskills.io/specification.md, https://github.com/agentskills/agentskills/discussions/252, https://github.com/anthropics/claude-code/issues/18837, https://github.com/anthropics/claude-code/issues/27113.

**Q6. Reverse references at script level in Kubernetes / Terraform / Bazel.** Answered in H5. Kubernetes stores `ownerReferences` on the dependent (forward-pointing to owner); garbage collector builds reverse index at runtime in-memory. Terraform has forward-only graph; destroy walks in reverse order. Bazel Sky Query introspects the Skyframe graph kept in server memory. Universal pattern: forward edges authoritative, reverse views computed. Sources: https://kube.rs/controllers/relations/, https://developer.hashicorp.com/terraform/internals/graph, https://bazel.build/reference/skyframe.

**Q7. Rename/deprecate cascade.** npm: `npm deprecate pkg "use X instead"` emits warnings at `npm install`. Consumers discover at next install or CI. No push notification. Cargo has no first-class `deprecated = true` in `Cargo.toml` (crates.io issue #549 open since 2017); the canonical pattern is a thin re-export wrapper crate. The "deprecation shim" pattern always breaks silently when (a) warnings are suppressed in dependency builds, or (b) transitive consumers never see the warning at all. kiho adopts the shim pattern with an additional `kb-lint` stale_reference check that forces migration debt into CEO visibility. Sources: https://docs.npmjs.com/cli/v11/commands/npm-deprecate/, https://github.com/rust-lang/crates.io/issues/549.

**Q8. 200+ skill scaling in AgentSkillOS.** The paper's hierarchical capability tree is purely a *discovery* structure, not a dependency structure. §2.1.1 describes building it bottom-up by LLM-driven categorization. §2.2.2 notes DAG-based orchestration satisfies forward-unidirectional constraints only. Skills are independent nodes; cross-skill references inside bodies are not modeled. AgentSkillOS solves discovery, not dependency tracking. kiho's CATALOG.md is architecturally equivalent and similarly silent on dependencies. Source: https://arxiv.org/html/2603.02176v1.

**Q9. Negative results — rejected approaches.** (a) Declarative project-level dependencies: closed "not planned" (#27113). (b) Static analysis of skill bodies for behavioral/dependency extraction: arXiv 2604.02837 §4 explicitly rejects — "no static analysis tool can fully characterize the behavioral scope of natural language instructions." (c) Signature field in frontmatter: rejected in agentskills RFC #252. (d) `allowed-tools` as a real permission boundary: #18837 reports it is not enforced. (e) Embedding-based similarity as default retrieval: every paper that measured accuracy at scale uses tree-based hierarchical routing. Sources as above.

**Q10. Merge primitive for combining two artifacts preserving both strengths.** Answered in H4. Git has no clean primitive. `ort`/`recursive` auto-resolve where diffs don't conflict; `-X ours`/`theirs` drop one side in every contested hunk. Octopus merge refuses any merge requiring manual resolution. Microsoft's "manual octopus merging" devblog presents the pattern as a clever hack, not a primitive. Conclusion: merging is a human act with mechanical inputs, not a mechanical act. Sources: https://git-scm.com/docs/merge-strategies, https://devblogs.microsoft.com/oldnewthing/20190515-00.

## What was deliberately NOT added

1. **Precomputed reverse-dependency index on disk** — H5, universal pattern rejection. Walk forward edges on demand.
2. **Embedding-based similarity search** — H1/Q2, requires daemon, non-deterministic across model versions, non-explainable. K-Dense-AI/claude-skills-mcp is an MCP daemon for precisely this reason.
3. **Mechanical "merge two skills" primitive** — H4, literature-confirmed open problem. Provide overlap inputs; humans write the third artifact.
4. **Top-level `requires:` or `dependencies:` frontmatter field** — H2, upstream rejection of #27113 and precedent from #252. Everything under `metadata.kiho.*`.
5. **AST parsing of skill bodies for semantic dep extraction** — arXiv 2604.02837 §4 rejects the category outright. Regex for wiki-link mentions is fine; prose intent extraction is not.
6. **Dependency lockfile** — there is exactly one version of each skill on disk. Lockfile would be pure bureaucracy.
7. **Auto-rename cascade** — renaming is a CEO committee decision. Tools assist, do not automate.
8. **Retroactive `requires:` annotation of existing 37 skills** — `kiho_rdeps` works without forward declarations by falling back to agent portfolios, CATALOG `parent_of`, and wiki-link scans. Annotation happens lazily.
9. **Mean-pairwise-Jaccard catalog-wide CI gate** — tempting per H3 §5.3 but risks retroactively invalidating existing skills. v5.15 tracks the metric via `--catalog-health` mode as informational only. v5.16 may elevate it to a gate if confusability climbs.
10. **Cross-plugin dependency tracking** — Claude Code issue #9444 (plugin-to-plugin deps) is still open upstream with no decision. kiho stays within its own skill tree.

## Recommended patterns adopted

1. **Forward-only dependency declarations under `metadata.kiho.*`** — five fields: `requires`, `mentions`, `reads`, `supersedes`, `deprecated`. Spec-compliant under agentskills.io `metadata:` slot.
2. **Reverse references computed on demand via `kiho_rdeps.py`** — zero on-disk cache, forward-edge scan across six sources.
3. **MinHash/Jaccard pre-create similarity gate** — Jaccard on unigrams + bigrams at block threshold 0.60, warn threshold 0.30.
4. **Semantic confusability tracking via `--catalog-health` mode** — informational only in v5.15, gate-eligible in v5.16.
5. **Deprecation shim pattern from npm/cargo** — in-place body rewrite to one-paragraph redirect; consumer migration lazy via `skill-improve`.
6. **Hierarchical CATALOG.md as discovery source of truth** — already done in v5.14 routing block.
7. **Human-driven merge with mechanical overlap inputs** — Gate 17 provides the Jaccard + shared tokens + unique-to-each samples; humans use `skill-improve` or `skill-derive`.
8. **Gate 17 `--force-overlap` override as CEO committee decision point** — not a retry button; logged per use to `skill-invocations.jsonl`.
9. **`kb-lint` stale_reference check as migration debt tracker** — advisory by default; escalates to exit 1 at 5+ findings to force CEO attention.

## Patterns explicitly rejected

1. **Precomputed reverse index on disk.** See H5 and NOT-added #1.
2. **Embedding similarity with on-disk vector index.** See Q2 and NOT-added #2.
3. **AST parsing of SKILL.md bodies.** See Q9 and NOT-added #5.
4. **Repurposing `allowed-tools` as a dep declaration.** Not enforced (#18837); conflates permissions with composition.
5. **Top-level `requires:` outside `metadata.kiho.*`.** Collides with future spec. See H2.
6. **Mechanical skill merge primitive.** See H4 and NOT-added #3.
7. **Catalog-level dependency lockfile.** See NOT-added #6.
