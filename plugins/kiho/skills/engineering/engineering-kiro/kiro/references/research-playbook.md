# 研究手册（按需触发）

## 核心原则

研究不是必经阶段，而是按需工具。先查 steering 文档和代码库，只在信息不足时触发研究。

## Fast Path（跳过研究的条件）

满足以下 **全部** 条件时，跳过研究直接进入规格：

1. ✅ Steering 文档已覆盖技术栈和模式选择
2. ✅ 代码库中有类似功能的参考实现
3. ✅ 无需引入新库或新模式
4. ✅ 需求清晰，无重大歧义
5. ✅ 无需竞品/行业调研

> 如果项目的 steering 文档完善，大多数功能应走 Fast Path。代码库 + steering 通常已提供足够上下文。

---

## 何时触发研究

### 需求阶段

- **陌生领域**: 涉及从未接触的业务逻辑（如地理围栏、支付流程、推送通知）
- **竞品模式**: 用户提及"像美团的 X"或"类似 DoorDash" — 调研其 UX 模式
- **合规要求**: 涉及用户数据、位置、支付、内容审核的功能

### 设计阶段

- **多方案并存**: 2+ 架构方案且无明显赢家（如 REST vs tRPC，URL state vs Zustand）
- **需要新库**: 功能需要已批准栈中没有的包
- **性能关键**: 有严格性能要求（实时更新、大列表、图片处理）
- **集成问题**: "X 库是否支持项目的框架版本 / 运行时 / 渲染模式？"

### 实现阶段（罕见）

- **技术阻塞**: 意外错误、兼容性问题、文档缺失
- **模式发现**: 实现中发现更好的模式，可能改变方案

---

## 研究请求格式

触发 `kiro-researcher` 时，使用以下结构：

```
QUESTION: [具体问题]

CONTEXT: [哪个规格阶段，指导什么决策，为什么重要]

CODEBASE_SCOPE:
- Search: [glob 模式、目录、文件名]
- Check: [需要验证的具体文件或模式]
- Steering: [相关的 .kiro/steering/ 文档]

EXTERNAL_SCOPE:
- Libraries: [需要调研的具体库]
- Patterns: [需要查找的设计模式或最佳实践]
- Sources: [偏好的文档来源，如有]
```

## 研究集成协议

### 单研究员（默认）

适用于有明确答案的直接问题：

1. Spawn kiro-researcher（QUESTION + CONTEXT + SCOPE）
2. 等待研究报告
3. 审查发现
4. 如有清晰建议 → 纳入规格
5. 如不清晰 → 向用户呈现选项，或 spawn 第二研究员（→ 辩论协议）

### 并行研究员（辩论协议）

适用于有多个有效方案的重大决策。详见 `debate-framework.md`。

## 质量检查清单

将研究发现纳入规格前：

- [ ] **代码库一致**: 发现不与 `.kiro/steering/` 规则冲突
- [ ] **版本兼容**: 库/模式与当前栈版本兼容（参照 `.kiro/steering/` 中的技术栈定义）
- [ ] **有据可查**: 每条结论引用具体来源（文件路径、URL、文档页面）
- [ ] **权衡明确**: 利弊都有记录，不只展示积极面
- [ ] **可决策**: 发现被框定为可执行选项，不是信息堆砌
- [ ] **用户决策标记**: 任何剩余歧义明确标记为需用户输入

## 反模式

- **过度研究**: 不要研究代码库已有答案的问题。先查 steering。
- **兔子洞**: 设定范围限制。超出范围时报告已有信息并标记缺口。
- **过时来源**: 验证库版本。2022 年关于 React 17 的博客可能不适用。
- **意见当证据**: 区分社区共识、官方文档和个人观点。
- **跳过代码库**: 没有代码库调查的外部研究是无用的 — 永远先查本地。
